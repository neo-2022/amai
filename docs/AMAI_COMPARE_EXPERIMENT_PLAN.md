# Amai Compare Experiment Plan

Этот документ не является master-plan всей памяти `Amai`.

Его роль:
- модульный план по user-facing сравнению `с Amai` / `без Amai`;
- частный roadmap внутри общего документа [AMAI_GLOBAL_MEMORY_ROADMAP.md](AMAI_GLOBAL_MEMORY_ROADMAP.md).

По глобальной цепочке внедрения этот план входит прежде всего в:
- Этап 7 `Compare and benchmark plane`;
- опирается на более ранние truth/scopes/memory этапы.

## Цель

Сделать в human dashboard (панель для человека-оператора) два честных режима сравнения:

- `онлайн` (живая история реальных ответов);
- `бенчмарк` (отдельный прогон заранее заданного набора тестов).

В каждом из этих режимов должны быть две реальные линии:

- `с Amai`
- `без Amai`

Сравнение должно показывать реальную разницу по лимитам без потери качества, точности и правды ответов.

## Главные договорённости

- Это не VS Code-only (не только для VS Code) решение.
- Любая платформа, к которой подключается Amai, должна уметь отдавать данные в общий формат.
- VS Code/Codex не должен оставаться особым случаем в live-обнаружении агентов.
- Правильный источник живых агентов должен жить внутри Amai, а не в одном внешнем файле или журнале одной платформы.
- Режим `без Amai` не означает полный отрыв от Amai.
- Правильная модель: `Amai не помогает, но продолжает измерять`.
- `экономия_N%` остаётся отдельным механизмом и в этот эксперимент не вмешивается.
- `онлайн` и `бенчмарк` нельзя смешивать в один и тот же график или в одну и ту же сводку.

## Язык пользовательского интерфейса

Всё, что видит человек на dashboard и в карточке сравнения, должно быть на русском языке:

- названия режимов;
- названия кнопок;
- названия переключателей;
- подписи быстрых интервалов;
- всплывающие подсказки;
- подписи осей и легенды;
- статусы бенчмарка;
- summary (сводка) и детали по клику на точку.

Внутренние технические поля, storage keys (ключи хранения), API field names и internal identifiers (служебные имена внутри кода) могут оставаться техническими, если это нужно для реализации.
Но весь user-facing text (текст для пользователя) должен быть русским.

## Что уже зафиксировано из обсуждения

Ниже список решений, которые уже согласованы и не должны потеряться:

- сравнение должно быть platform-neutral, а не только для VS Code;
- любой runtime и любой провайдер могут вести как на локальные, так и на облачные модели, поэтому нельзя делать вывод по одному имени адаптера или провайдера;
- live-обнаружение активных агентов должно жить внутри Amai, а не в журнале одной платформы;
- пользователь видит две линии: `с Amai` и `без Amai`;
- один реальный ответ даёт одну точку и попадает только в одну линию, а не в обе сразу;
- режим `без Amai` означает: Amai не помогает, но продолжает измерять;
- у графика два режима: `онлайн` и `бенчмарк`;
- `онлайн` и `бенчмарк` не смешиваются ни в данных, ни в summary;
- в `бенчмарк` режиме есть отдельная кнопка запуска, отдельный dataset и отдельный run;
- benchmark-линии должны быть наложены друг на друга по общей нормализованной benchmark-временной шкале;
- для `онлайн` режима должно быть расширенное управление диапазоном времени;
- для `бенчмарк` режима такой time picker не нужен;
- по клику на точку должны открываться детали конкретного ответа;
- в деталях точки обязательны: время, агент, модель, платформа, `5ч KPI`, токены, лимиты, `thread_id`, `session_id`, `query type`;
- для benchmark-точек дополнительно обязательны `benchmark step / elapsed minute` и `benchmark case id`;
- всё, что видит пользователь в UI, должно быть на русском языке;
- лицевая сторона карточки должна по-прежнему показывать процент, а не словесные объяснения.

## Как показывать несколько агентов

Если одновременно работают несколько агентов, их нельзя бездумно сваливать в один и тот же compare-график.
Иначе получится каша, и сравнение `с Amai` / `без Amai` начнёт врать.

Правильное поведение такое:

- по умолчанию на графике показывается один выбранный агент;
- над онлайн-графиком есть переключатель `Агент`;
- если агентов несколько, пользователь выбирает, чей график смотреть;
- сравнение `с Amai` / `без Amai` строится только внутри выбранного агента;
- если у агента менялась модель, это тоже должно быть видно и фильтруемо;
- если у агента сильно смешаны разные типы запросов, нужен фильтр по `типу запроса`.

Что делать не надо:

- не надо в первой версии рисовать на одном графике сразу всех агентов поверх двух compare-линий;
- не надо смешивать точки разных агентов только потому, что у них одинаковая модель;
- не надо смешивать разные модели одного агента без явного фильтра.

Итоговое правило:

- основной compare-график в первой версии = один выбранный агент;
- multi-agent обзор можно добавить позже отдельным summary-блоком, но не как основной graph surface.

## Как правильно определять агентов на любых платформах

### Что должно считаться агентом

Агент должен определяться не по названию окна и не по тому, где мы его случайно увидели.

У агента должна быть нормальная живая карточка личности:

- платформа;
- внутренний `agent_scope` Amai;
- идентификатор чата / треда на стороне платформы;
- идентификатор сессии;
- модель;
- провайдер модели;
- рабочий проект или repo root (корень проекта), если он есть.

### Как это должно работать правильно

Каждая платформа должна присылать в Amai одинаковый live signal (живой сигнал о себе).

Простыми словами:

- VS Code/Codex пишет в Amai: `я такой-то агент, работаю в таком-то чате, на такой-то модели`;
- любая другая платформа с любым runtime должна писать в Amai то же самое;
- любая другая платформа тоже должна писать это в том же формате.

Тогда Amai определяет активных агентов не по локальному журналу одной программы, а по своему собственному единому live registry (реестру живых агентов).

### Что нельзя делать

- нельзя считать VS Code журнал главным источником истины для всех платформ;
- нельзя угадывать агента только по имени окна;
- нельзя смешивать двух разных агентов только потому, что у них одинаковая модель;
- нельзя смешивать два разных чата одного агента в одну live-сущность;
- нельзя делать какую-либо платформу, runtime или провайдера second-class citizen (второсортным клиентом).

### Какой должен быть общий live contract

Каждая платформа должна уметь отдавать в Amai минимальный обязательный набор:

- `platform_id` (кто прислал сигнал: VS Code, OpenClaw и т.д.);
- `platform_thread_id` (идентификатор чата/треда у этой платформы);
- `agent_scope` Amai;
- `session_id`;
- `agent_label` для человека;
- `model`;
- `runtime_adapter` (через какой runtime/API идёт вызов: OpenAI, Anthropic, Ollama, OpenAI-compatible gateway и т.д.);
- `model_provider` (чей это model/backend бренд, если он вообще отделим от runtime);
- `execution_mode` (где реально крутится модель: `local`, `cloud`, `hybrid`);
- `repo_root` или другой project binding (привязка к проекту), если есть;
- время последнего живого сигнала;
- флаг, что агент реально подключён к модели.

Если платформа не может отдать этот минимум, Amai не должна делать вид, что агент определён точно.
Такой runtime (живой источник) должен попадать только в слой `raw/pending binding` (сырой/ещё не связанный слой), а не в полноценные активные агенты.

### Что важно не перепутать

Нельзя считать, что одно имя `provider` или `runtime_adapter` автоматически определяет, где реально работает модель.

Это неверно, потому что один и тот же runtime или шлюз может работать по-разному:

- один и тот же runtime может вести на локальную модель;
- тот же runtime может вести на удалённую модель;
- аналогично и другие runtime/шлюзы могут проксировать как локальные, так и облачные модели.

Поэтому truth-model должна строиться не по одному слову `provider`, а по связке:

- `runtime_adapter`;
- `execution_mode`;
- `usage_truth_source`;
- `limit_truth_source`.

### Что это даст на примере платформы с внешним runtime

Если вы подключите новую платформу через любой runtime, правильная схема должна быть такой:

- платформа отправляет в Amai живые сигналы в том же формате, что и VS Code;
- Amai видит: это такая-то платформа, такой-то runtime, такая-то модель, такой-то chat/thread;
- отдельно Amai знает, это `local` или `cloud`, и не угадывает это только по имени runtime;
- ответы этой платформы начинают попадать в ту же общую статистику;
- на дашборде агент этой платформы виден на тех же правах, что и VS Code-агент;
- линии `с Amai` и `без Amai` продолжают считаться честно и одинаково.

То есть VS Code должен стать просто одним из клиентов Amai, а не хозяином всей live-логики.

## Что такое две линии

### Линия `с Amai`

Ответы, в которых Amai реально помогает:

- retrieval (поиск полезного контекста);
- память;
- continuity (восстановление рабочего контекста);
- prompt/context augmentation (добавление подсказки и контекста в запрос к модели).

### Линия `без Amai`

Ответы, в которых Amai не помогает, но остаётся наблюдателем:

- не добавляет память;
- не добавляет retrieval/context (поиск и подмешивание контекста);
- не меняет prompt (текст, который отправляется модели);
- не влияет на ответ модели;
- но продолжает записывать метрики и лимиты.

## Как собирается статистика

После каждого ответа модели создаётся одна точка статистики.

Одна точка = один реальный ответ.

Точка должна записываться всегда:

- и в линии `с Amai`;
- и в линии `без Amai`.

Точнее:

- если ответ был с помощью Amai, точка попадает в линию `с Amai`;
- если ответ был без помощи Amai, точка попадает в линию `без Amai`.

Никаких guessed/counterfactual (угаданных или гипотетических) значений для линии `без Amai` не строим.
Только реальные ответы и реальные лимиты после них.

## Что записывается в каждую точку

Для каждого ответа сохраняем:

- точное время ответа;
- платформа;
- агент;
- модель;
- `runtime_adapter` (через какой runtime/API шёл ответ);
- `execution_mode` (`local`, `cloud`, `hybrid`);
- принадлежность к линии `с Amai` или `без Amai`;
- exact `5ч KPI` (точный показатель 5-часового лимита) в момент после ответа;
- токены ответа;
- остаток лимитов после ответа;
- `usage_truth_source` (откуда взяли правду по расходу: `provider_usage`, `runtime_usage`, `local_estimate`);
- `limit_truth_source` (есть ли вообще внешний limit truth: `provider_limit` или `none`);
- `thread_id` (идентификатор треда / чата);
- `session_id` (идентификатор сессии);
- clean/dirty sample (чистая или загрязнённая точка);
- pressure/debug markers (служебные метки давления на лимит и отладки).

### Как считать лицевой процент на разных runtime

Лицевой `%` должен выглядеть одинаково для человека, но truth-source под ним может быть разный.

Правило такое:

- если есть реальный provider limit, `%` считаем от него;
- если provider limit нет, `%` считаем от реальной цены ответа в токенах;
- если runtime не дал точных токенов, `%` считаем от локальной оценки токенов;
- источник расчёта сохраняется в точке, но на лицевой стороне не расписывается словами.

Это нужно затем, чтобы:

- для OpenAI / Anthropic / Gemini с честным внешним limit source процент мог считаться по реальной шкале;
- для локальных моделей процент не превращался в фальшивый “лимит провайдера”, которого нет;
- для любого runtime local/cloud режим определялся честно, а не по названию адаптера.

Допустимые debug markers (служебные метки).

### Что реально нужно оставить в первой версии

- giant-thread pressure (раздутость большого чата);
- sample cleanliness (насколько точка чистая для сравнения);
- exact-pair availability (есть ли точная пара для сравнения).
- query type (тип запроса).

Почему:

- `giant-thread pressure` прямо помогает понять, что лимит жрёт сам раздутый чат, а не качество Amai;
- `sample cleanliness` помогает не сравнивать грязные точки, где в лимит вмешался ещё кто-то или что-то;
- `exact-pair availability` показывает, можно ли вообще честно делать вывод про реальную экономию.
- `query type` нужен, чтобы сравнивать похожие ответы с похожими, а не смешивать в одной картине, например, багфикс, поиск по коду и onboarding.

### Что можно оставить только как вторичный слой

- context pressure (давление размером контекста);
- foreign activity suspicion (подозрение на постороннюю активность);
- answer-like marker (похоже ли событие на нормальный ответ);

Почему:

- `context pressure` частично уже виден из токенов ответа и заполненности окна, поэтому это полезно, но не обязательно как отдельная главная метка;
- `foreign activity suspicion` лучше включать как пояснение причины для `sample cleanliness`, а не как отдельный крупный показатель;
- `answer-like marker` полезен для внутренних расчётов, но если точка графика уже создаётся только по реальным ответам, для пользователя это вторично;

## Как должен выглядеть график

График живёт на обороте карточки `Экономия токенов за текущую сессию`.

Существующий double-click flip (двойной клик для переворота карточки) сохраняется.

У графика должно быть два режима просмотра:

- `онлайн`;
- `бенчмарк`.

Эти режимы используют разные наборы данных и разные сводки.

### Оси

- для режима `онлайн`: X = реальное время ответа;
- для режима `бенчмарк`: X = нормализованное benchmark-время (например, минута от старта прогона или общий номер benchmark-case), чтобы две benchmark-линии накладывались друг на друга;
- Y = значение KPI.

### Точки и линии

- каждая точка = один ответ;
- следующая точка = следующий ответ;
- точки соединяются линией;
- одна линия показывает ответы `с Amai`;
- вторая линия показывает ответы `без Amai`.

Для режима `онлайн` обе линии строятся только по реальным рабочим ответам.
При этом online-график по умолчанию строится для одного выбранного агента, а не для всех сразу.

Для режима `бенчмарк` обе линии строятся только по benchmark-run (отдельному прогону тестового набора), а не по обычной рабочей истории.
При этом в режиме `бенчмарк` линии должны быть наложены друг на друга по общей benchmark-временной шкале, а не разнесены по разному wall-clock времени двух последовательных прогонов.

Режим не нужно дублировать текстом внутри точки.
Он уже понятен по самой линии.

## Что должно открываться по клику на точку

Клик по точке открывает детали именно этого ответа.

Обязательные поля:

- точное время ответа;
- benchmark elapsed minute / benchmark step, если точка из benchmark-режима;
- benchmark case id, если точка из benchmark-режима;
- агент;
- модель;
- платформа;
- exact `5ч KPI` в этот момент;
- сколько токенов занял ответ;
- сколько оставалось лимитов;
- `thread_id`;
- `session_id`;
- clean/dirty sample;
- pressure/debug markers.

Если есть данные, дополнительно можно показывать:

- query type (тип запроса);
- exact pair status (состояние точной пары для сравнения);
- `with_amai` / `without_amai` measured tokens (измеренные токены с Amai и без Amai);
- answer-like marker (признак, что это именно ответ).

### Минимально обязательный состав tooltip в первой версии

- точное время ответа;
- агент;
- модель;
- платформа;
- exact `5ч KPI` в этот момент;
- сколько токенов занял ответ;
- сколько оставалось лимитов;
- `thread_id`;
- `session_id`;
- `sample cleanliness`;
- `giant-thread pressure`;
- `exact-pair availability`.
- `query type`.

### Какие benchmark-кнопки нужны в первой версии

На странице `Бенчмарк` в первой версии нужны именно эти кнопки:

- группа `Экономия`:
- `Живая экономия на одном ответе`
  - контур: `./scripts/proof_token_live_turn_savings_matrix.sh`
  - что мерит: насколько Amai реально уменьшает расход на одном ответе в нескольких типовых shape-сценариях;
  - короткая подсказка: `Показывает, насколько Amai уменьшает расход лимита на одном ответе.`

- `Живая экономия на continuity`
  - контур: `./scripts/proof_token_art_live_turn_savings.sh`
  - что мерит: насколько Amai экономит лимит на реальном continuity-запросе, а не только на synthetic форме;
  - короткая подсказка: `Показывает, как Amai экономит лимит на реальном continuity-сценарии.`

- `Набор запросов на экономию`
  - контур: `./scripts/proof_token_benchmark_suite.sh`
  - что мерит: среднюю экономию на наборе запросов, а не на одном удачном примере;
  - короткая подсказка: `Показывает среднюю экономию токенов на наборе типовых запросов.`

- группа `Качество`:
- `MCP-задачи агента`
  - контур: `./scripts/proof_mcp_task_matrix.sh`
  - что мерит: выдерживает ли Amai agent-задачи с tool routing, hostile fail-closed и isolation;
  - короткая подсказка: `Показывает, как Amai справляется с задачами агента и не путает tool и scope.`

- `Память и изоляция`
  - контур: `./scripts/proof_memory_task_matrix.sh`
  - что мерит: сохраняет ли Amai память честно и не течёт ли она между агентами и проектами;
  - короткая подсказка: `Показывает, держит ли Amai память правильно и не смешивает разные контексты.`

- `Навыки и память действий`
  - контур: `./scripts/proof_procedural_benchmark.sh`
  - что мерит:
    - рождаются ли полезные skills из опыта;
    - не плодятся ли дубли навыков;
    - правильно ли skill переиспользуется в похожей задаче;
    - не проходит ли плохой skill в `verified`;
    - не всплывает ли stale skill как лучший способ;
    - умеет ли evaluator тормозить плохую эволюцию procedural memory;
  - короткая подсказка: `Показывает, умеет ли Amai превращать опыт в полезные навыки и не тащить плохие навыки дальше.`
  - текущий materialized metric bundle:
    - `reuse quality` через `proof_procedural_seed` и `proof_procedural_shadow_review`;
    - `bad-skill suppression` через `proof_negative_procedural_memory`;
    - `stale-skill suppression` через `proof_skill_refinement_contour` и `proof_skill_version_history`;
    - `shadow-to-verified uplift` через `proof_procedural_shadow_review` и `proof_shared_promotion_by_approval`;
    - `evaluator correctness` через `proof_shared_promotion_by_approval` и fail-closed части остальных procedural proofs.

### Какие benchmark-семейства уже есть как ориентиры

Это не обязательно отдельные кнопки первой версии, но их надо держать в плане как следующий слой:

- `LiveMCPBench` и `MCP-Universe`
  - уже частично materialized через локальную MCP task matrix;
- `AgentBench`
  - полезен как ориентир для multi-environment agent benchmark;
- `AssistantBench`
  - полезен как ориентир для длинной работы агента и resume after interruption;
- `FACTS Grounding`
  - полезен как следующий quality-gate, чтобы не продавать одну только экономию без проверки grounded answers;
- `SWE-bench Verified`
  - полезен как следующий coding-agent слой после запуска базового compare/benchmark режима.
- отдельный `procedural memory benchmark`
  - обязателен как следующий контур именно для skill reuse, bad-skill suppression и quality of procedural evolution.

### Важное правило для benchmark-страницы

Не все benchmark-кнопки должны рисовать один и тот же смысл графика.

Правильно так:

- группа `Экономия`
  - показывает, как меняется расход и KPI;
- группа `Качество`
  - показывает, проходит ли система задачи, не ломает ли память и не теряет ли изоляцию.

Для procedural benchmark внутри группы `Качество` график должен уметь показывать не один общий score, а выбранную метрику:
- reuse quality;
- bad-skill suppression;
- stale-skill suppression;
- shadow-to-verified uplift;
- evaluator correctness.

То есть:

- benchmark-страница одна;
- поле графика одно;
- но смысл Y-оси и summary зависит от выбранного benchmark-а.

Если выбран benchmark из группы `Экономия`, график показывает экономию/перерасход.
Если выбран benchmark из группы `Качество`, график показывает метрику качества или прохождения задач.
Это должно быть явно подписано над графиком, чтобы человек не путался.

### 0. Platform-neutral live agent front-door (общий вход для живых агентов)

Online compare surface и benchmark compare surface должны храниться и рендериться отдельно.

## Как сделать бенчмарк честным

Чтобы benchmark не врал, нужны простые правила:

- оба прогона должны идти на одном и том же benchmark dataset;
- агент, модель и runtime должны быть одинаковыми внутри одного benchmark-run;
- порядок прогонов должен сохраняться в run passport;
- порядок лучше чередовать между запусками:
  - один раз сначала `с Amai`;
  - другой раз сначала `без Amai`;
- если benchmark чувствителен к прогреву или кэшу, это должно быть видно в run passport;
- если fairness нарушена, benchmark не должен молча выглядеть “честным”.

Это нужно, чтобы второй прогон не получал скрытое преимущество только потому, что первый уже что-то прогрел.

## Как внедрять без накопления ошибок

Делать нужно не так:

- сначала написать всё подряд;
- потом в самом конце идти и разгребать большой ком ошибок.

Делать нужно так:

1. Сделали один значимый шаг.
2. Сразу прогнали его тесты.
3. Сразу сделали ручную проверку UI и API именно этого шага.
4. Сразу исправили найденные проблемы.
5. Только после этого перешли к следующему шагу.

Это обязательное правило для этой задачи.

### Обязательный цикл после каждого значимого шага

После каждого значимого шага должен быть один и тот же цикл:

- unit tests, если шаг затронул внутреннюю логику;
- route/API tests, если шаг затронул `observe` или dashboard payload;
- ручная проверка на живом dashboard;
- короткий список найденных недостатков;
- исправление этих недостатков;
- повторная проверка;
- только потом переход к следующему шагу.

### Что считать значимым шагом

Значимый шаг — это любой шаг, после которого пользователь уже может увидеть новое поведение или после которого может сломаться честность сравнения.

Например:

- новый state;
- новые payload поля;
- новые фильтры;
- новая страница `Бенчмарк`;
- новая benchmark-кнопка;
- новый график;
- новый tooltip;
- новый fairness guard.

### Отдельное правило для UI

Если шаг меняет интерфейс, одной только автоматической проверки недостаточно.
После такого шага обязательно нужны:

- ручной просмотр в dashboard;
- проверка, что русский текст читается нормально;
- проверка, что кнопки и переключатели не конфликтуют друг с другом;
- проверка, что карточка не стала перегруженной;
- проверка, что старое поведение не сломалось.

### Отдельное правило для benchmark fairness

Если шаг меняет benchmark-run, порядок прогонов, кэш, прогрев или benchmark passport, нельзя откладывать проверку на потом.
Такой шаг нужно проверять сразу отдельным коротким proof-циклом, иначе потом уже будет непонятно, где именно появилась нечестность сравнения.

## Реализация по шагам

### Шаг 1. Сделать platform-neutral live registry агентов

Ввести единый live front-door внутри Amai для обнаружения активных агентов.

Что должно появиться:

- единый формат heartbeat/identity event для платформ;
- Amai-owned registry живых агентов;
- отдельный адаптер для VS Code/Codex;
- отдельные адаптерные пути для внешних платформ/runtime;
- слой `pending binding`, если платформа прислала неполные данные.

Проверка после шага:

- активные агенты определяются через Amai registry, а не напрямую через `.codex/state_5.sqlite`;
- VS Code продолжает правильно определяться;
- любую новую платформу и любой runtime можно подключить по тому же контракту;
- один и тот же агент не дублируется от повторных heartbeat;
- неполные платформенные сигналы не притворяются “полноценным активным агентом”.

### Шаг 2. Ввести experiment state (состояние эксперимента)

Сделать platform-neutral state (общее состояние для любой платформы) для двух линий:

- `with_amai`;
- `without_amai_but_measuring`.

Проверка после шага:

- состояние сохраняется;
- состояние переживает refresh;
- `экономия_N%` не ломается;
- состояние одинаково доступно для VS Code и любых других платформенных адаптеров;
- выбранный агент и фильтры онлайн-графика тоже сохраняются;
- существующие target-selector tests (тесты переключателя целевого KPI) зелёные.

### Шаг 3. Сделать runtime bypass (режим без помощи Amai)

В режиме `без Amai` выключить помощь Amai, но не измерение.

Проверка после шага:

- ответ идёт без Amai augmentation (без подмешивания контекста и памяти);
- observability (наблюдение) остаётся активной;
- лимиты и токены всё ещё снимаются;
- режим работает одинаково через любой платформенный адаптер, а не только через VS Code;
- sample всё ещё записывается.

### Шаг 4. Добавить per-answer sample (одну точку на один ответ)

Записывать sample (точку статистики) после каждого ответа модели.

Проверка после шага:

- один ответ создаёт одну точку;
- несколько ответов создают несколько точек;
- у точки есть стабильная платформенная идентичность агента, а не только display name (отображаемое имя);
- у каждой точки есть время, агент, модель, платформа и KPI.
- точки разных агентов не смешиваются без явного выбора пользователя.

### Шаг 5. Построить compare surface (подготовленный набор данных для графика)

Собрать две реальные серии и summary по ним.

Проверка после шага:

- серия `с Amai` содержит только ответы с Amai;
- серия `без Amai` содержит только реальные ответы без Amai;
- guessed values (угаданные значения) не появляются;
- точки от любых платформ читаются одним и тем же compare surface;
- compare surface умеет строиться по одному выбранному агенту;
- API (интерфейс данных для фронта) возвращает обе линии и summary (сводку).

### Шаг 6. Добавить benchmark mode (режим отдельного прогона)

Добавить отдельный benchmark path (контур бенчмарка), который запускается кнопкой на обороте карточки.

Что должно появиться:

- кнопка `Бенчмарк` на странице `Онлайн`;
- кнопка `Онлайн` на странице `Бенчмарк`;
- отдельная страница `Бенчмарк` на обороте карточки;
- поле графика для выбранного benchmark-run;
- набор benchmark-кнопок для первой версии:
  - `Живая экономия на одном ответе`
  - `Живая экономия на continuity`
  - `Набор запросов на экономию`
  - `MCP-задачи агента`
  - `Память и изоляция`
  - `Навыки и память действий`;
- у каждой benchmark-кнопки есть русская краткая подсказка по hover;
- заранее определённый benchmark dataset;
- два последовательных прогона: `с Amai`, потом `без Amai`, или наоборот;
- отдельный `benchmark_run_id`;
- отдельные benchmark-линии и benchmark-summary;
- run passport с конфигурацией benchmark-run;
- статус выполнения benchmark-run в UI.

Проверка после шага:

- со страницы `Онлайн` по кнопке открывается страница `Бенчмарк`;
- на странице `Бенчмарк` та же кнопка меняет надпись на `Онлайн` и возвращает назад;
- benchmark стартует по кнопке конкретного benchmark-а;
- у каждой benchmark-кнопки hover показывает русскую подсказку;
- online-история не смешивается с benchmark-историей;
- во время benchmark-run сначала растёт одна линия, потом вторая;
- по завершении есть две benchmark-линии для честного сравнения;
- две benchmark-линии наложены друг на друга по общей benchmark-временной шкале;
- run passport сохраняет порядок прогона и конфигурацию;
- если benchmark прервался, это видно как `failed`, а не как тихо сломанный график.
- procedural benchmark не рисуется как generic memory score:
  - должно быть явно видно, какую именно procedural метрику сейчас показывает график.
- если у procedural benchmark пока materialized только линия `с Amai`, UI обязан показывать partial benchmark-state явно:
  - `without Amai` не дорисовывается guessed-линией;
  - карточка и snapshot обязаны явно сказать, что second lane ещё `not_materialized`.
- после materialization честной линии `without_amai_but_measuring` compare-plane обязан перейти в dual-line state:
  - `without Amai` идёт через explicit bypass-run, где Amai не помогает, но продолжает измерять;
  - карточка больше не имеет права оставаться в ложном `pending`, если обе benchmark-линии уже реально измерены.

### Шаг 11. Live verification (живая ручная проверка)

Сделать ручную живую проверку:

- benchmark не загрязняет online-историю;
- `экономия_N%` не сломан.
