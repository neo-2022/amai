# Maintainability / Supportability / Evolvability / Anti-Hardcoding

## 0. Статус документа

Настоящий раздел обязателен для всех подсистем проекта.
Он задаёт требования к организации кода, модулей, зависимостей, контрактов, конфигурации, сопровождения, восстановления и безопасного изменения системы.

Нарушение любого обязательного правила из этого раздела считается:

- архитектурным дефектом;
- дефектом сопровождения;
- дефектом change safety;
- при необходимости — блокером релиза.

## 1. Назначение

Цель этого стандарта — гарантировать, что система остаётся:

- понятной;
- изменяемой;
- тестируемой;
- выпускаемой;
- откатываемой;
- восстанавливаемой;
- доказуемо сопровождаемой;
- не только для автора кода, но и для другого инженера через месяцы и годы после написания.

Проект считается хорошо сопровождаемым только в том случае, если другой инженер может:

- понять источник истины;
- безопасно найти нужную зону;
- локально изменить поведение;
- запустить правильные проверки;
- выпустить изменение;
- откатить его;
- восстановить систему;
- объяснить, что произошло и почему.

## 2. Область действия

Стандарт обязателен для:

- Core;
- Agent;
- Console / Browser / UI;
- Contracts / Schemas;
- Storage;
- Replay / Determinism;
- Policy;
- Observability / Audit / Evidence;
- Release / Rollback / Recovery;
- Security / Privacy / Redaction;
- Tooling / CI / Gates;
- Support / Diagnostics / Admin tooling.

## 3. Базовые определения

### 3.1 Maintainability

Способность системы безопасно изменяться, исправляться и развиваться без деградации качества.

### 3.2 Supportability

Способность системы быть понятной, диагностируемой, сопровождаемой, обслуживаемой и восстанавливаемой в эксплуатации.

### 3.3 Evolvability

Способность системы принимать новые требования, режимы и интеграции без переписывания архитектурного основания.

### 3.4 Source of Truth

Единственный канонический источник истины для конкретного словаря, правила, контракта, политики, матрицы или данных.

### 3.5 Truth

Канонические подтверждённые данные, на которых строится принятие решений.

### 3.6 Projection

Derived/local/view state: кэш, read-model, группировка, визуализация, UI state, индекс, spatial layout, экспортное представление и иные производные формы.

### 3.7 Invariant

Правило, которое должно сохраняться при любом рефакторинге, деградации, восстановлении и релизе.

### 3.8 Hardcoding

Жёсткое вшивание в код изменяемого правила, словаря, матрицы, environment-specific значения, tenant-specific поведения или policy, которые должны жить в source-of-truth, config, contracts, registry или policy layer.

## 4. Базовый принцип

Обязательно:

- единицей проектирования считать доменный модуль с инвариантами, а не файл;
- проектировать систему по:
  - доменным границам;
  - инвариантам;
  - источникам истины;
  - направленным зависимостям;
  - режимам изменений;
  - режимам отказа и восстановления.

Запрещено:

- принимать архитектурные решения по правилам вида:
  - "файл не должен быть длиннее N строк";
  - "функция обязана помещаться на экран";
  - "любой большой файл надо срочно дробить";
  - "любой маленький файл автоматически хороший".

Критерий готовности:

Для любой критической зоны можно ответить:

- за что она отвечает;
- где её источник истины;
- какие инварианты она обязана сохранять;
- кто её владелец;
- какие проверки обязательны;
- как её откатить/восстановить.

## 5. Декомпозиция по доменам

Обязательно:

- делить проект по смысловым доменам и инвариантам;
- разделять минимум следующие зоны:
  - domain/truth;
  - contracts;
  - policy;
  - storage;
  - transport;
  - streaming;
  - audit/proof;
  - replay/determinism;
  - projection/UI;
  - privacy/redaction;
  - release/rollback/recovery;
  - runtime guardrails;
  - external integrations.

Запрещено:

- смешивать в одном модуле:
  - domain;
  - transport;
  - storage;
  - policy;
  - UI;
  - infra glue;
  - release logic;
  - recovery logic.

Критерий готовности:

- каждый критический модуль можно описать одной доменной фразой;
- для него существует явная граница:
  - входы;
  - выходы;
  - побочные эффекты;
  - зависимости;
  - тестовые обязательства.

## 6. Размер файлов и функций

Обязательно:

- использовать размер файла и функции только как повод для анализа;
- анализировать длинные файлы и функции на:
  - смешение ответственностей;
  - скрытые побочные эффекты;
  - плохую тестируемость;
  - размытые preconditions/postconditions;
  - чрезмерные зависимости.

Запрещено:

- автоматически считать длинный файл дефектом;
- автоматически считать короткий файл качественным;
- дробить код так, чтобы разрушалась причинно-следственная логика и видимость инвариантов.

Критерий готовности:

- для любого большого файла есть архитектурное объяснение, почему он остаётся цельным;
- если объяснения нет, он разбит по доменным инвариантам.

## 7. Направленные зависимости

Обязательно:

- поддерживать направленные зависимости между слоями;
- стремиться к следующей схеме:
  - truth/domain выше projection;
  - policy выше transport/UI projection;
  - storage скрыт за интерфейсами;
  - replay/determinism независимы от presentation;
  - UI не является источником истины;
  - transport не задаёт доменные правила.

Запрещено:

- циклические зависимости между доменами;
- протекание storage/UI/transport деталей в domain;
- использование projection как скрытого truth layer.

Критерий готовности:

- любую критическую зависимость можно объяснить архитектурно;
- нет циклических связей между ключевыми подсистемами.

## 8. Truth и Projection

Обязательно:

- явно различать truth и projection;
- любое derived/local/UI state маркировать как projection, пока оно не подтверждено каноническим источником;
- всегда указывать происхождение критически важных данных.

Запрещено:

- рендерить stale/local/derived данные как confirmed truth;
- использовать projection для критического решения без явной валидации;
- скрывать от оператора/разработчика происхождение состояния.

Критерий готовности:

Для каждого критичного отображаемого состояния можно ответить:

- это truth или projection;
- как подтверждается;
- где source of truth.

## 9. Policy enforcement

Обязательно:

- держать каноническое enforcement policy в backend/domain слое;
- разрешать UI только проецировать policy для удобства;
- делать fail-closed, если policy-check невозможен, неполон или устарел.

Запрещено:

- делать UI единственным местом проверки;
- кэшировать policy так, чтобы stale UI открывал запрещённое действие;
- размазывать policy-логику по случайным handlers и helper-функциям.

Критерий готовности:

- любое опасное действие проверяется канонической policy независимо от UI;
- невозможно выполнить запрещённое действие только из-за устаревшей проекции.

## 10. Ownership

Обязательно:

- у каждой критической зоны должен быть owner;
- для каждой зоны должны быть определены:
  - owner кода;
  - owner тестов;
  - owner документации;
  - owner runbooks;
  - owner release/rollback риска;
  - owner recovery semantics.

Запрещено:

- критические зоны без owner;
- размытая ответственность "этим занимаются все";
- ситуации, когда документация и код живут без согласованного владельца.

Критерий готовности:

- для любой критической зоны можно однозначно назвать owner.

## 11. Места поиска

Обязательно:

- организовать проект так, чтобы было заранее ясно:
  - где contracts;
  - где incident typing;
  - где policy;
  - где observability/gap registry;
  - где runbooks;
  - где replay/determinism;
  - где migrations;
  - где release rules;
  - где privacy/redaction;
  - где backup/recovery;
  - где tests по доменам.

Запрещено:

- размазывать одну и ту же логику по нескольким несистемным местам;
- иметь несколько неявно конкурирующих источников истины;
- держать критические знания только "в голове команды".

Критерий готовности:

- новый инженер может найти нужную область без устных подсказок.

## 12. Документация для сопровождения

Обязательно:

- поддерживать документацию, отвечающую на вопросы:
  - где это меняется;
  - какие инварианты нельзя ломать;
  - какие тесты обязательны;
  - какие runbooks относятся;
  - какой rollback path;
  - какие known failure modes существуют.
- иметь один master source-of-truth документ и не плодить параллельные канонические версии одной и той же истины.

Запрещено:

- документация "про всё вообще", не помогающая безопасно менять код;
- устная передача критических правил как основной механизм поддержки;
- несинхронные документы, расходящиеся с кодом и контрактами.

Критерий готовности:

По документации можно безопасно:

- найти нужную зону;
- понять риски;
- определить обязательные тесты;
- выполнить rollback/recovery.

## 13. Тестируемость как свойство архитектуры

Обязательно:

- для каждой критической зоны заранее определять обязательные типы тестов:
  - unit;
  - contract;
  - integration;
  - e2e;
  - replay/regression;
  - chaos/fault injection;
  - load/soak;
  - drill/tabletop/pentest.
- проектировать модули так, чтобы тесты были естественны и локализованы.

Запрещено:

- считать критическую логику "нормально покрытой", если она проверяется только вручную;
- держать модуль в форме, где тестирование требует несоразмерного окружения без причины;
- полагаться на unit-тесты там, где нужен contract/integration/e2e/replay/drill.

Критерий готовности:

- для каждой critical зоны можно перечислить обязательные тесты и их местоположение.

## 14. Тестовые слои

Обязательно:

- Unit — на локальные правила, канонизацию, валидацию, преобразования, классификацию, redaction.
- Contract — на API, schemas, policy, incident typing, truth modes, config sources.
- Integration — на auth, policy, storage, stream, replay, KMS, external boundaries.
- E2E — на пользовательские и операторские сценарии.
- Replay/regression — на детерминизм и воспроизводимость.
- Chaos/fault injection — на отказ зависимостей, деградацию и восстановление.
- Load/soak — на давление и долговременную стабильность.
- Drill/tabletop/pentest — на эксплуатационную готовность.

Запрещено:

- заменять один тестовый слой другим "для удобства";
- объявлять critical path готовым без слоя, соответствующего риску.

Критерий готовности:

- критические инварианты защищены тестами нужного уровня.

## 15. Ошибки и деградация

Обязательно:

- типизировать ошибки и отказные состояния;
- описывать degraded mode:
  - когда включается;
  - что сохраняет;
  - что отключает;
  - как из него выйти.
- для critical failures иметь recovery path и runbook.

Запрещено:

- магические деградации без явного условия входа/выхода;
- ошибки без контекста;
- скрытые fallback-и, ослабляющие гарантии без сигнала.

Критерий готовности:

При критическом сбое можно определить:

- что произошло;
- как классифицировано;
- что система сохранила;
- как восстановиться.

## 16. Observability и supportability

Обязательно:

- проектировать observability для будущего разбора инцидентов;
- должна быть возможность восстановить:
  - путь события/запроса;
  - применённую policy;
  - причину отказа;
  - причину degraded mode;
  - источник incident classification;
  - происхождение evidence.

Запрещено:

- логирование "для автора", а не для сопровождения;
- потеря критических сигналов под давлением без отдельного аварийного режима;
- логирование секретов, токенов, паролей, ключей, неотрезанных чувствительных данных.

Критерий готовности:

- по observability можно доказуемо объяснить поведение системы в инциденте.

## 17. Повторяемость инженерных решений

Обязательно:

- использовать единые словари и форматы:
  - incident types;
  - severity;
  - truth modes;
  - evidence refs;
  - audit refs;
  - gap naming;
  - policy identifiers;
  - release/profile markers;
  - privacy impact markers.
- похожие задачи решать одинаково.

Запрещено:

- авторские локальные паттерны без общей модели;
- несколько несовместимых схем одной и той же сущности;
- дублирование одной и той же логики разными способами.

Критерий готовности:

- для типовой задачи существует канонический способ реализации.

## 18. Запрет хардкодинга

Обязательно:

- все изменяемые правила, матрицы, словари, профили, incident types, feature flags, policy mapping, platform matrix, retention rules, severity mapping, routing rules, trusted domains, supported versions, build profiles, privacy/redaction logic и иные управляемые проектом значения должны задаваться через:
  - contracts;
  - config;
  - source-of-truth файлы;
  - schema/registry;
  - policy definitions;
  - typed constants — только если это архитектурный invariant.
- любое значение, которое:
  - может измениться;
  - зависит от окружения;
  - зависит от tenant/profile/mode;
  - уже фигурирует в документации, contracts, registry, checklist или config,
  обязано браться из канонического источника.

Запрещено:

- хардкодить в логике приложения:
  - incident codes;
  - platform support matrix;
  - tenant-specific behavior;
  - environment-specific URLs, limits, flags;
  - privacy/redaction/retention/legal-hold rules;
  - release/canary/rollback routing;
  - trusted identities/domains;
  - crypto profiles;
  - policy decisions,
  если для этого существует или должен существовать source-of-truth;
- дублировать одно и то же правило в нескольких местах;
- прятать изменяемые правила в `if/else`, `switch/match`, helper-функции, UI, tests без связи с каноническим источником.

Допустимо:

- hardcode только:
  - фундаментальные архитектурные константы;
  - compile-time invariants;
  - protocol-level fixed markers;
  - значения, которые по определению не настраиваются и не зависят от среды.

Критерий готовности:

- для каждого критичного правила можно показать один канонический источник истины;
- изменение правила не требует ручного поиска по нескольким несвязанным местам;
- любой remaining hardcode либо:
  - осознанный invariant,
  - либо дефект архитектуры.

## 19. Технический долг

Обязательно:

- типизировать технический долг как:
  - documented;
  - contracted;
  - implemented;
  - tested;
  - gated;
  - drill-exercised;
  - production-enforced;
  - planned;
  - deprecated;
  - temporary.
- явно отмечать временные и частично реализованные решения;
- привязывать долг к owner и плану закрытия.

Запрещено:

- скрытый технический долг;
- временные решения без явной маркировки;
- ложное ощущение завершённости там, где есть только документация.

Критерий готовности:

- для любой critical зоны можно честно определить её maturity/status.

## 20. Рефакторинг

Обязательно:

- рефакторить модуль, если:
  - он меняется по нескольким причинам;
  - смешивает домены;
  - трудно тестируется;
  - инварианты не видны локально;
  - одна правка тянет каскад;
  - критическая логика спрятана в glue/helper слоях.
- сохранять при рефакторинге:
  - contracts;
  - observability;
  - replay determinism;
  - audit/proof;
  - rollback/recovery semantics;
  - privacy invariants.

Запрещено:

- рефакторинг ради внешней красоты, ухудшающий supportability;
- отложенный рефакторинг "потому что пока работает", если структура уже взрывает стоимость изменений;
- дробить код до потери причинно-следственной целостности.

Критерий готовности:

После рефакторинга:

- границы яснее;
- тестируемость не хуже;
- change impact локальнее;
- rollback/recovery не сложнее.

## 21. Onboarding и bus factor

Обязательно:

- строить систему так, чтобы новый инженер мог безопасно войти в неё без устной "тайной традиции";
- иметь:
  - master source-of-truth;
  - единые naming rules;
  - owner-зоны;
  - checklist master;
  - карту тестовых слоёв;
  - changelog/ADR/исторические решения.

Запрещено:

- критические участки, понятные только автору;
- неявные правила, существующие только в голове команды;
- критические изменения без письменного контекста.

Критерий готовности:

- новый инженер может по структуре, документам и тестам начать безопасно сопровождать проект.

## 22. Release / Rollback / Recovery как часть maintainability

Обязательно:

- считать release safety, rollback safety и recoverability частью сопровождаемости кода;
- для каждого рискованного изменения понимать:
  - путь выката;
  - условия остановки;
  - rollback criteria;
  - влияние на evidence/audit/replay/privacy;
  - post-deploy и post-restore проверки.

Запрещено:

- рассматривать сопровождение как "только организацию кода";
- игнорировать release/rollback/recovery в архитектурной оценке;
- считать restore успешным, если потеряна доказуемость, replay-детерминизм, audit integrity или truth labeling.

Критерий готовности:

- любое critical изменение имеет release path, rollback path, recovery path и обязательные post-checks.

## 23. Fail-safe инварианты сопровождения

Обязательно:

Следующие инварианты не могут нарушаться ни при каких обстоятельствах:

- источник истины не подменяется projection;
- policy-as-UI не является единственным enforcement;
- dangerous actions fail closed;
- stale/local cache не отображается как confirmed truth;
- backup без проверенного restore не считается backup;
- recovery, теряющий evidence integrity / proof / replay determinism, не считается успешным;
- критические observability-сигналы сохраняются под давлением;
- tenant isolation не ослабляется в degraded mode;
- секреты не попадают в логи, UI, экспорт, evidence bundles;
- audit chain остаётся проверяемой;
- AI-недоступность может ухудшить удобство, но не корректность;
- release/rollback/recovery не ломают source-of-truth semantics;
- детерминизм critical replay path сохраняется после рефакторинга, миграции и восстановления;
- dual-write mismatch после допустимого окна доставки должен быть равен нулю.

Запрещено:

- любой degraded mode, который расслабляет:
  - tenant isolation;
  - policy enforcement;
  - truth labeling;
  - audit/proof integrity;
  - privacy/redaction guarantees.

Критерий готовности:

- для каждого fail-safe инварианта существуют:
  - формулировка;
  - owner;
  - test/gate/drill;
  - incident/recovery semantics.

## 24. Обязательный change-review

Обязательно:

Перед принятием значимого изменения команда обязана ответить:

- какой домен меняется;
- какой источник истины затронут;
- какие инварианты рискуют сломаться;
- какие тесты обязательны;
- какой rollback path;
- как это влияет на audit / evidence / replay / privacy / observability / recovery;
- какие docs / checklists / runbooks / contracts нужно обновить;
- кто owner результата;
- не создаёт ли изменение новый hardcode;
- не появляется ли новый скрытый technical debt.

Запрещено:

- считать изменение готовым без прохождения этого review path.

Критерий готовности:

- на каждое critical изменение есть проверяемый change-review след.

## 25. Метрики сопровождаемости

Обязательно:

- отслеживать минимум:
  - lead time change;
  - mean time to understand hotspot;
  - rollback time;
  - mean time to reproduce incident;
  - долю изменений, затронувших больше доменов, чем планировалось;
  - flaky test rate;
  - долю critical paths без drills;
  - процент модулей без owner;
  - количество мест правки для одного логического изменения;
  - documentation drift;
  - number of source-of-truth violations / hardcoding violations.

Запрещено:

- оценивать maintainability только "по ощущениям";
- игнорировать повторяющиеся сигналы о плохой сопровождаемости.

Критерий готовности:

- сопровождаемость измеряется наблюдаемыми показателями, а не интуицией.

## 26. Финальный критерий соответствия

Система считается соответствующей этому стандарту только если одновременно выполняются все условия:

- source-of-truth явен;
- projection не подменяет truth;
- policy enforcement каноничен;
- critical зоны имеют owner;
- change impact локализуем;
- обязательные тесты очевидны и существуют;
- release/rollback/recovery paths определены;
- observability поддерживает доказуемое сопровождение;
- документация помогает безопасно менять;
- hardcoding изменяемых правил отсутствует;
- технический долг видим и типизирован;
- новый инженер может безопасно войти в проект;
- fail-safe invariants формализованы и защищены;
- поддержка не держится на памяти одного автора.

## 27. Запрещающая формула

Проект не считается хорошо сопровождаемым, если выполняется хотя бы одно из условий:

- источник истины неочевиден;
- projection может подменить truth;
- policy живёт только в UI;
- критические зоны без owner;
- change impact нельзя локализовать;
- обязательные тесты неочевидны;
- rollback/recovery неформализованы;
- observability не позволяет доказуемо объяснить инцидент;
- hardcoding заменяет модель данных/контракты/конфиг;
- технический долг скрыт;
- новый инженер не может безопасно начать сопровождение без устной передачи знаний.

## 28. Каноническая формула стандарта

Хорошо сопровождаемый проект — это проект, в котором другой инженер через месяцы или годы может без устной "тайной традиции" понять источник истины, локально и безопасно изменить нужный домен, запустить правильные проверки, выпустить изменение, откатить его при необходимости, восстановить систему после сбоя и доказуемо объяснить, что произошло — без подмены truth projection'ом, без скрытого хардкодинга и без распада инвариантов.
